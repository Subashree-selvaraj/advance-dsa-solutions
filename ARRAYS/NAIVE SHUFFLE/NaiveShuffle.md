# Question 1.A — The In-Place Shuffle  
## Part i_i

### Question

**Provide a rigorous logical or mathematical proof detailing why a naive iterative loop executing from index 0 to n-1 utilizing the expression `swap(arr[i], arr[rand() % n])` inherently introduces systemic statistical bias. Contrast the total number of unique execution paths generated by this naive execution tree against the actual cardinality of the true permutation set.**

---

## Naive shuffle under discussion

The naive approach uses the following idea:

```text
For each index from 0 to n-1,
pick a random index from 0 to n-1,
then swap the two elements.
```

This method appears random, but it is not a correct uniform shuffle.

---

## Why the naive shuffle is biased

The fundamental issue is that the algorithm always chooses a random index from the **entire array**, even for elements that were already processed.

This causes:

- already processed positions to be modified again
- elements to move multiple times
- some final permutations to be reachable through more paths than others

As a result, the final distribution is not uniform.

---

## Execution path count

At every iteration:

- there are `n` possible random choices
- the loop runs for exactly `n` iterations

Therefore, the total number of unique execution paths is:

```text
n × n × n × ... × n = n^n
```

So:

```text
Total naive execution paths = n^n
```

---

## True permutation count

For `n` distinct elements, the total number of actual permutations is:

```text
n!
```

So:

```text
Total true permutations = n!
```

---

## Why this creates bias

For a shuffle to be uniform, all permutations must receive the same number of execution paths.

That would require:

```text
n^n / n!
```

to be an integer.

But in general, for `n ≥ 3`, this is not true.

Therefore:

- paths cannot be distributed equally among all permutations
- some permutations must receive more execution paths
- those permutations become more likely

Hence the naive shuffle is statistically biased.

---

## Example for n = 3

For a 3-element array:

```text
Naive execution paths = 3^3 = 27
Actual permutations   = 3!  = 6
```

Now:

```text
27 / 6 = 4.5
```

This is not an integer.

That means 27 paths cannot be evenly divided among 6 permutations.

So some permutations are generated by 4 paths and others by 5 paths.

Thus the probabilities are unequal.

---

## Structural reason for the bias

The algorithm processes the array from left to right:

```text
i = 0, 1, 2, ..., n-1
```

But at every step it chooses:

```text
j from 0 to n-1
```

So when a position is already processed, it can still be changed later.

Example:

- position 0 may be swapped when `i = 0`
- it may be swapped again when `i = 1`
- again when `i = 2`
- and so on

This means no position is ever truly fixed.

That repeated interference is what destroys uniformity.

---

## Comparison with the correct method

| Feature | Naive Shuffle | Fisher–Yates |
|---|---|---|
| Loop direction | Forward | Backward |
| Random range | Always `0..n-1` | Shrinks to `0..i` |
| Positions fixed permanently | No | Yes |
| Total execution paths | `n^n` | `n!` |
| True permutation count | `n!` | `n!` |
| Uniform output | No | Yes |

---

## Mathematical conclusion

Let:

- `S` = set of all execution paths of the naive shuffle
- `P` = set of all permutations of `n` elements

Then:

```text
|S| = n^n
|P| = n!
```

A uniform shuffle requires each permutation to correspond to the same number of paths.

That is impossible unless:

```text
|S| / |P| = integer
```

Since this does not hold in general, the naive shuffle cannot produce a uniform distribution.

Therefore, it is inherently biased.

---

## Key takeaway

### One-line summary

```text
Forward loop + full random range every step = biased shuffle
```
